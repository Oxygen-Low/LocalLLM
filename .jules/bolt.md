## 2024-04-05 - Safe OnPush Optimization for ngModel Components
**Learning:** Adding `ChangeDetectionStrategy.OnPush` to Angular components that heavily use Signals is a major performance win, but it's crucial to verify how it interacts with traditional bindings. In this codebase, complex components (like `settings.page.ts`) mix Signals for some state with standard properties bound via `[(ngModel)]` for form inputs. I initially worried `OnPush` might break `[(ngModel)]` since the underlying properties aren't Signals. However, because `[(ngModel)]` inherently fires an event (`ngModelChange`) on every user input, Angular's `OnPush` change detection is correctly triggered anyway.
**Action:** When identifying Signal-heavy components to optimize with `OnPush`, do not be deterred by the presence of `[(ngModel)]` bindings. They are safe to mix and will not break under `OnPush` because user events automatically trigger the necessary view updates.
## 2026-04-07 - [Optimize Component Rendering with OnPush Change Detection]
**Learning:** In a highly interactive app that leverages Angular Signals for reactive state management, omitting `ChangeDetectionStrategy.OnPush` on complex UI components leads to unnecessary re-renders driven by default change detection, degrading performance and increasing CPU consumption, especially during heavy interactions like streaming updates.
**Action:** Default to `ChangeDetectionStrategy.OnPush` for presentational or signal-driven component decorators. This keeps Angular’s change detection targeted and helps avoid redundant component checks, but components that rely on mutable inputs, imperative subscriptions, or external non-signal state may still require `ChangeDetectionStrategy.Default` or explicit `markForCheck()`.
